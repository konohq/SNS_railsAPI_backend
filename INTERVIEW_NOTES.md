# 面接説明用メモ

## 30秒の概要

Rails APIとReact SPAで作成したX風のミニSNSです。JWT認証を使い、投稿、リプライ、いいね、フォロー、リポスト、プロフィール画像変更を実装しました。特に、画面上の制御だけに頼らず、`current_user`を基準にAPI側で所有権を検証した点と、重複操作をDB制約でも防ぐ設計を意識しました。開発後半では、テストとCIを使って既存機能を守りながらリファクタリングしています。

## 技術的にアピールできる点

- Rails APIとReact SPAの分離
- Devise + JWTによる認証
- `current_user`基準の認可
- User同士の自己参照多対多によるフォロー
- Post自己参照によるリポスト
- Active Storageによる画像アップロード
- PostgreSQLのunique indexと外部キー
- N+1を意識したpreload
- RSpec、RuboCop、Brakeman、GitHub Actions

## 強みの説明例

「機能数だけでなく、APIを直接呼ばれた場合にも他人のデータを変更できないことを意識しました。例えば投稿削除では、IDだけで検索せず`current_user.posts.find`として、操作対象をログインユーザーの所有物に限定しています。また、いいねやフォローの重複はvalidationだけでなくDBの一意制約でも防ぎ、同時リクエストでもデータが壊れにくい設計に改善しました。」

## 苦労した点

### フォローの自己参照関連

回答例:

「User同士の関係なので、通常の1対多より命名と関連が分かりにくい点に苦労しました。中間テーブルに`follower_id`と`followed_id`を持たせ、User側で能動関係と受動関係を分けました。実装後に、モデルvalidationだけでは重複を完全に防げないことに気づき、複合unique indexとrequest specも追加しました。」

### JWTログアウト

回答例:

「最初はブラウザのtokenを削除すればログアウトになると考えていましたが、それだけでは盗まれたtokenが期限まで使えると分かりました。そこでDevise JWTのrevocation endpointを呼び、サーバー側でもJTIを更新するようにしました。この改善から、UI上の状態とサーバー上の認証状態を分けて考える重要性を学びました。」

### N+1

回答例:

「投稿一覧では投稿ごとにユーザー、いいね、コメント、フォロー状態を表示するため、serializer内の参照でN+1が起きやすい構造でした。SQLログを確認し、必要な関連をpreloadするとともに、一覧APIへ本当に必要な情報だけを返すよう見直しました。」

## 改善した経験として話せる内容

「初期実装ではReactの`App.jsx`に画面とAPI処理を集めて、まず動くものを完成させました。しかし機能追加で420行まで増え、変更影響を追いにくくなりました。そこで先にAPIのrequest specを追加し、その後、投稿、認証、プロフィール、フォローの単位でコンポーネントとhooksを分割しました。見た目を変えずに責務だけを分け、各PRを小さくしたことで、既存機能を壊さず改善できました。」

## 「なぜRails API + Reactにしたのですか」

回答例:

「フロントとバックエンドの責務を分け、実務でよくあるJSON APIを介した構成を学びたかったためです。Rails側では認証、認可、validation、DB整合性を担当し、React側では非同期通信とUI状態を担当しました。構成が複雑になるデメリットはありますが、API契約やCORS、認証tokenの扱いまで学べる点を重視しました。」

## 「なぜJWTにしたのですか」

回答例:

「SPAからAuthorization headerでAPIを呼ぶ構成を経験したかったためです。一方、localStorageはXSS時にtokenを取得されるリスクがあるので、短い有効期限、ログアウト時の失効、CSP、依存管理が必要だと認識しています。本番要件によってはHttpOnly Cookie方式も候補にし、その場合はCSRF対策とセットで判断します。」

## 「なぜService Objectを使わないのですか」

回答例:

「現時点では単純なCRUDが多く、すべてをService Objectにするとファイル数と追跡コストだけが増えると判断しました。複数モデルをまたぐ処理、外部API連携、transactionが必要な処理が増えた時点で導入します。現在はコントローラ、モデル、serializerの責務を明確にすることを優先しています。」

## 「なぜReduxを使わないのですか」

回答例:

「現規模では共有状態が主にログインユーザーと投稿一覧なので、Contextとcustom hooksで十分だと判断しました。状態管理ライブラリは、画面数や共有状態が増えて更新経路が複雑になった時に導入します。技術を増やすこと自体ではなく、複雑さを下げるかで判断しています。」

## 「なぜこのDB設計にしたのですか」

回答例:

「フォローはUser同士の多対多なのでRelationshipを中間テーブルにし、フォローする側とされる側を明確にしました。リポストは元投稿を参照するPostとして表現しています。いいね、フォロー、リポストの重複はアプリケーションのvalidationに加えてDBのunique indexでも防ぎ、競合時の整合性を守っています。」

## 「セキュリティで意識したことは何ですか」

回答例:

「認証済みかだけでなく、そのユーザーが対象データを操作してよいかをAPI側で確認しました。Strong Parameters、CORS origin限定、JWT失効、画像の種類とサイズ制限、秘密情報の環境変数管理も行っています。また、BrakemanとDependabotをCIに組み込み、公開後も継続的に確認できるようにしています。」

## 「今後どう改善しますか」

回答例:

「優先順位は、新機能より品質です。まず認証・認可のrequest specとDB制約を完成させ、CIでバックエンドとフロントエンドの両方を検証します。次にAPIレスポンスを統一し、Reactの単一コンポーネントを機能別に分割します。その後、TypeScript、E2Eテスト、画像ストレージのクラウド対応を段階的に進めます。」
